LINE Developers Console Wrapper

Diagnosis of the current console + proposed unified hierarchy view (cookie-authenticated, read-only).

1. Diagnosis — current console experience

Navigated developers.line.biz/console/ via Chrome DevTools. Without a live session it redirects to the LINE Business ID login (account.line.biz/login). The signed-in experience is structured as follows (per official docs + project domain model):

  1. Providers — top-level page. Each row is a provider (id, name, certified flag, region). Click a provider to drill in.
  2. Provider page → Channels tab — list of channels under the provider, tagged by type: Messaging API LINE Login LINE MINI App Blockchain Service
  3. Channel page → multiple tabs: Basic settings Messaging settings LINE Login settings LIFF — credentials, webhook, callback URLs, status spread across tabs.
  4. LIFF apps — nested under the LIFF tab of a LINE Login channel: liffId, view {type (compact/tall/full), url}, description.
Pain point: viewing one provider + one channel + one tab costs 3+ page navigations and tab switches. Credentials and status are never visible at the same time. There is no single at-a-glance view of the whole hierarchy.

2. Proposed — unified expandable hierarchy list

One row per hierarchy level, behind a search + refresh toolbar. Expand a provider → provider info + nested channels (each channel type is a distinct badge). Expand a channel → channel info at a glance + nested LIFF apps. Secrets are masked by default with clickable reveal & copy. Status uses an icon alongside color, never color alone. Reuses the look of this project's existing line-account-hierarchy component. Three variants below — A dense card tree, B an IDE-style tree viewer, and C a master–detail split. Headers are reordered so the type badge sits right after the name, with the id and status trailing; the details block no longer repeats the name.

3. Data model & cookie approach

Hierarchy types

  • ConsoleProvider — id, name, region, certified, createdAt
  • ConsoleChannel — channelId, type, name, status + type-specific fields
  • ConsoleLiffApp — liffId, view{type,url}, description
A single ChannelType = messaging | login | miniApp | blockchain discriminated union tags each channel row — drives the avatar color and which nested info block renders.

Cookie-based adapter

A LineConsoleAdapter (read-only) injected with a developer-console cookie header. Default endpoints hit developers.line.biz; the fetch and base URL are overridable so a same-origin proxy can sidestep CORS.

  • listProviders()
  • listChannels(providerId)
  • getChannel(channelId)
  • listLiffApps(channelId)

4. Risks & open questions

Recommended deployment: Manifest V3 browser extension. A dedicated standalone page on a non-LINE domain is blocked by cookie scope (httpOnly + SameSite on .line.biz) and missing CORS headers. A browser extension with host_permissions for https://developers.line.biz/* solves both: its background/content fetch is treated as first-party — the existing session cookie is attached automatically and CORS is not enforced.
Extension (MV3) primary target — pass the extension's own fetch into the adapter, baseUrl stays the default. Backend proxy fallback for non-extension hosts — adapter points at a proxy baseUrl.
The web components here are still plain custom elements; an extension just hosts them in a side panel / popup and injects the cookie-authenticated adapter. They remain usable in any page that supplies an adapter.
Endpoint stability. The internal console REST API is undocumented and may change. Without a live session I modeled endpoints from the documented console structure; exact paths need to be confirmed against a real signed-in network capture and wired via adapter config.
Read-only scope. This wrapper focuses on at-a-glance viewing (list + reveal secrets), matching your ask. Create/edit/delete stay in the real console to avoid duplicating mutating, undocumented endpoints.
Q: secret reveal & copy? Should secrets show on expand (masked by default, click-to-reveal + copy), and should LIFF view-type be editable inline or link out to the console? Default plan: reveal-only, with deep-links back to the console page for edits.

5. Implementation location

To live in src/web/developers-console/:

  • types.ts — console provider / channel / LIFF view types
  • console-adapter.ts — createLineConsoleAdapter (pluggable fetch + baseUrl; cookie forwarded in extension context)
  • line-developers-console.ts — the expandable hierarchy web component
  • define.ts + index.ts — registration & public exports
Matches the existing line-account-* Lit + adapter conventions in src/web/, reusing the same CSS variables and avatar/pill styling.
